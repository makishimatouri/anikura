# 可复用方法与计算规则

这份文档记录当前代码已经使用或维护时必须保持一致的规则。计算结果必须同时给出输入、公式、中间值和各个结果；示例不是线上数据。

## 1. 活动公开资格

当前公开查询的基本条件是：

```text
公开活动 P = review_status == "approved" 且 status == "ongoing"
```

不同页面再叠加自己的条件：

```text
精选活动 F = P 且 is_featured == true
AniROX 活动 A = P 且 is_anirox == true
即将发生活动 U = P 且 date >= 当前日期
```

维护时要分别报告：

```text
公开活动数 = 满足 P 的记录数
精选活动数 = 满足 F 的记录数
AniROX 活动数 = 满足 A 的记录数
即将发生数 = 满足 U 的记录数
```

不要把“已审核”“进行中”“精选”“AniROX”当作同一个标签。若要统计数据库数据，必须写出查询时间、时区、空值处理和去重方式。

## 2. 日期和月份筛选

### 月份边界

活动列表的月份筛选使用闭区间：

```text
monthStart = YYYY-MM-01
monthEnd = YYYY-MM-该月最后一天
```

代码用 `new Date(year, month, 0).getDate()` 取得月份最后一天，避免把 2 月或 9/11 月硬写成 31 号。

示例：输入 `2026-02`。

```text
输入：year = 2026，month = 2
中间值：new Date(2026, 2, 0).getDate() = 28
结果 1：monthStart = 2026-02-01
结果 2：monthEnd = 2026-02-28
校验：2026 不是闰年，因此 2 月有 28 天
```

### 时区注意

当前签到和部分活动查询使用：

```ts
new Date().toISOString().split("T")[0]
```

这取的是 UTC 日期。业务时区是 Asia/Shanghai，在北京时间凌晨到 UTC 日期切换之间可能出现边界差异。任何修复都要先写测试用例，分别验证：北京时间 00:05、UTC 前一天 16:05、月末和闰年；不能只改一处字符串。

## 3. 精选推荐随机选择

推荐列表先由 Supabase 查询筛选，再在前端选择索引：

```text
输入：候选数 n，随机数 r，0 <= r < 1
公式：index = floor(r × n)
结果范围：index ∈ {0, 1, ..., n - 1}
边界：n = 0 时不允许取模或访问候选项
```

示例：`n = 4`、`r = 0.625`。

```text
中间值：r × n = 0.625 × 4 = 2.5
结果 1：floor(2.5) = 2
结果 2：选择第 3 个候选项（数组从 0 开始）
校验：2 在 0 到 3 的合法索引范围内
```

当前首页精选条件还包括已审核、进行中、`is_featured = true`。候选为空时桌面轮播会显示演示数据，移动端会显示暂无精选提示；这是现有行为，不要在维护文档中当作真实活动统计。

## 4. 每日签到积分

当前代码的单次签到规则：

```text
earned = floor(random × 16) + 5
bonus = 10，当 newStreak >= 7 且 newStreak % 7 == 0；否则为 0
本次入账 = earned + bonus
新总积分 = 旧总积分 + earned + bonus
```

范围计算：

```text
random 的下界：0 <= random < 1
random × 16 的范围：0 <= random × 16 < 16
floor 后的整数：0 至 15
earned 的结果：5 至 20 积分，含两端
bonus 的结果：0 或 10 积分
本次入账的结果：5 至 30 积分
```

示例计算（不是线上结果）：旧总积分 `128`、上次连续签到 6 天、这次 `random = 0.625`。

```text
newStreak = 6 + 1 = 7
earned = floor(0.625 × 16) + 5 = floor(10) + 5 = 15
bonus = 10，因为 7 >= 7 且 7 % 7 = 0
本次入账 = 15 + 10 = 25
新总积分 = 128 + 25 = 153
结果 1：随机签到积分 = 15
结果 2：连续签到奖励 = 10
结果 3：本次流水 = 25
结果 4：新总积分 = 153
```

校验时要检查同一用户当天不能重复签到、`checkins`、`point_transactions` 和 `profiles.total_points` 是否一致。当前实现是多次客户端写入，不是一个数据库事务；如果要修复并发或重复入账问题，应单独设计 RPC/事务并做数据修复方案。

## 5. 积分兑换和库存

优惠券兑换的账务关系：

```text
兑换后积分 = 兑换前积分 - reward.points_cost
兑换后库存 = 兑换前库存 - 1（仅当库存不是 null）
积分流水 = -reward.points_cost
```

每次报告要分开写：

```text
输入：兑换前积分、兑换成本、兑换前库存
结果 1：兑换后积分
结果 2：兑换后库存，或“无限库存/未设置库存”
结果 3：积分流水负数
结果 4：兑换记录 code 是否生成
校验：兑换前积分 >= 成本；库存为 null 时不做数值扣减
```

当前前端先读余额再写兑换记录、余额、流水和库存。不要把前端检查当作并发安全保障；如果库存或积分出现异常，应先冻结相关运营入口并核对流水，再决定是否改代码或补数据。

## 6. 图片大小、路径和展示

当前上传代码限制：

```text
文件字节数 <= 5 × 1024 × 1024 = 5,242,880 bytes
```

结果分别是：

- 最大允许字节数：`5,242,880 bytes`。
- 按十进制换算：`5,242,880 / 1,000,000 = 5.24288 MB`。
- 按二进制换算：`5,242,880 / 1,048,576 = 5 MiB`。

上传路径：

```text
头图：headers/<时间戳>.<扩展名>
海报：posters/<时间戳>.<扩展名>
bucket：posters
```

头图推荐 16:9 或 21:9；海报详情页保持完整比例。改图片展示时要分别验证：原图比例、横向卡片裁切、空 URL 回退、Supabase 域名白名单和移动端加载。

## 7. 发布检查结果的写法

`npm run check` 当前依次执行：

```text
typecheck = npm run typecheck
lint = npm run lint
build = npm run build
```

报告不要只写“检查通过”，要分别写：

```text
typecheck：通过 / 失败，错误数
lint：通过 / 失败，error 数、warning 数
build：通过 / 失败，是否生成路由
线上 smoke test：逐个列出 URL 和 HTTP 状态
```

当前已知的非阻塞提示是 2 个 `<img>` warning，以及 Next.js 对 `middleware` 文件约定的弃用提示。warning 不是 error，但发布报告要保留，避免下次误以为是新问题。

## 8. 版本归档方法

版本不是计算机自动推断出来的事实，必须由变更范围决定：

- 新功能或明显页面行为变化：功能小版本，如 `v0.1.5`。
- 小修复、文案、样式或局部交互：子修订，如 `v0.1.51`。
- 阶段性大功能包：次版本，如 `v0.2.0`。
- 重大架构变化或公开稳定版：主版本，如 `v1.0.0`。

版本报告必须分别写：上一个 tag、当前 commit、用户可见变化、数据库变化、是否需要环境变量、验证结果、回滚点。完整命令见 `docs/VERSIONING.md`。
